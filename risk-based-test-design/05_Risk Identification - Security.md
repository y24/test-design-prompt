# PROMPT 05: リスク抽出（セキュリティ / 権限・監査・プライバシー）
PROMPT 00（rbt_min_v1）に準拠してJSONのみを出力せよ。
このプロンプトは「セキュリティ・権限・監査・プライバシー」観点でリスクを抽出する専門チェックパスである。

## 1) 目的
- 仕様/設計/手順/過去不具合/非機能要件から、セキュリティ事故・不正アクセス・情報漏えい・監査不備につながるリスクを洗い出す。
- 根拠がない断定は禁止。根拠が弱い場合は status="info" と info_requests に落とす。

## 2) スコープ（逸脱防止）
- PROMPT 01で確定した scope.in_scope のみ対象。
- scope.delta_scope.mode="diff_only" の場合：
  - touched_components と diff_summary に直接関係する認証/認可/データ露出/ログのみを見る。
  - 差分外の一般的なセキュリティ講義は禁止（気持ちは分かるが）。

## 3) 出力セクション（このプロンプトで出すもの）
- risk_register.risks（必須）
- risk_register.duplicates（可能なら）
- info_requests（根拠不足があれば必須）
※ test_catalog は出さない（ここではテスト設計に踏み込まない）

## 4) 抽出ガイド（セキュリティに特化）
優先的に探すリスクパターン：
- 認証（AuthN）：
  - セッション固定、トークン期限/失効不備、MFA要件漏れ、パスワード/リセットフローの穴
- 認可（AuthZ）：
  - 直リンク/IDOR（参照ID差し替え）、テナント境界漏れ、権限昇格、管理機能の露出
- データ露出：
  - 個人情報/機密の過剰表示、ログ/エラーメッセージへの漏えい、バックアップ/エクスポートの漏れ
- 入力/境界（代表的脆弱性クラス）：
  - インジェクション（SQL/OS/テンプレ）、XSS、CSRF、SSRF、パストラバーサル、アップロード
  ※ただし、根拠なく「脆弱です」と断定しない。設計上の入力点・エスケープ方針・権限境界などの根拠を示す。
- 監査・ログ：
  - 監査ログ未記録、改ざん耐性不足、追跡不能（誰が何をしたか残らない）、時刻/相関ID不足
- 暗号・秘密情報：
  - 平文保存、鍵管理不備、設定にシークレット直書き、暗号化範囲の誤り
- 運用・権限管理：
  - ロール設計不明、権限棚卸し不可、退職者権限残存、最小権限の逸脱

## 5) リスクの書き方（最小フォーマット厳守）
risk_register.risks[] の各要素は最低限これを満たす：
- risk_id: "RSK-0001" 形式（新規採番）
- title: 事故の姿が見える題名（例：「テナント境界漏れで他社データ閲覧」）
- category: 原則 "security"（監査/法令寄りなら compliance でも可）
- statement: 「もしXなら、Yの不正/漏えい/監査不備が起きる」形式で具体化
- priority: P0〜P3（根拠に基づく）
- status: confirmed/hypothesis/info
- evidence: source_id + locator + note（必須）
推奨：
- affected_assets: 画面/API/ロール/データ種別（PII等）/ログ/設定など
- recommended_tests: 文字列配列（例：権限マトリクス、IDOR、CSRF、入力バリデーション、監査ログ確認、秘密情報スキャン）
- canonical_key: 重複束ねキー
- extras: 自由欄（例：脅威モデル視点、境界/信頼境界、対象ロール、規制要件、保護レベル）

## 6) 根拠（evidence）の書き方（セキュリティ向け）
- 「入力点」「権限境界」「データ分類」「ログ要件」のいずれに紐づく根拠かを evidence.note に明記する。
- “記載がない”ことを根拠にする場合：
  - status は hypothesis か info
  - info_requests で「どの仕様/設定/実装を確認すべきか」を具体化する。

## 7) 重複（duplicates）
- 例：テナント境界漏れ/IDOR/権限昇格が表現違いで複数出たら canonical_key で束ねる。
- duplicates に canonical_key / risk_ids / merge_note を入れる（可能なら）。

## 8) info_requests を必ず出すべき典型
- ロール/権限表（権限マトリクス）がない or 不完全
- 個人情報/機密情報の定義（データ分類）が不明
- 監査ログ要件（記録対象、保管、閲覧権限、改ざん耐性）が不明
- セッション管理（期限、失効、同時ログイン、デバイス）不明
- 入力検証/エスケープ/CSRF対策方針が不明
- エクスポート/バックアップの保護方針が不明

## 9) 出力（JSONのみ）
- PROMPT 00のトップレベルに従い、contract/output_type/meta/scope/sources は必ず含める。
- risk_register（risksは必須）と、必要なら info_requests を出す。
- 余計なトップレベルキーを増やさない（extras に寄せる）。
