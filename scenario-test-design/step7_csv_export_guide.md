# step7_csv_export_guide.md

## 目的

このステップの目的は、Step6 までで確定した優先順位付きテストケースを、指定されたカラム構成の **CSV出力用データ** として整形することである。

このステップは、新たな観点やシナリオを追加する工程ではない。  
また、テストケース内容を再設計する工程でもない。  
ここで行うのは、既にレビュー・承認された内容を、CSVとして利用しやすい形へ整えることに限定する。

---

## このステップで扱う入力

このステップでは、主に以下を入力として扱う。

- Step5 の成果物（テストケース一覧）
- Step6 の成果物（優先順位付きテストケース一覧）
- ユーザーが指定したCSVカラム構成
- ユーザーが追加で提示した出力ルール
- 各ステップのレビューで反映された修正内容

CSV出力の主な起点は Step6 の成果物とする。  
priority を含まない旧版ではなく、直近で承認された最新版を基準とすること。

---

## このステップの成果物

このステップでは、**CSV出力用データ** を作成する。

基本カラム構成は以下とする。

- `scenario_id`
- `scenario_name`
- `priority`
- `preconditions`
- `steps`
- `expected_results`
- `test_data`

ユーザーが別のカラム順や追加カラムを明示した場合は、その指示を優先すること。

---

## このステップでやること

### 1. 直近確定版のテストケースを採用する
Step6 までで確定した内容を基準に、CSV出力対象を整理する。

### 2. 指定カラムへ値を対応づける
各テストケースの項目を、CSVの各列へ対応づける。

### 3. 複数行情報をCSVセル表現に整える
`steps` や `expected_results` など複数行になりやすい情報は、CSVセル内で破綻しない形式へ整える。

### 4. 改行・カンマ・引用符の扱いを整理する
CSVとして読み取りやすいよう、文字列整形ルールを適用する。

### 5. 出力前に列崩れしないか確認する
見た目だけでなく、CSVデータとして扱ったときに列がずれないようにする。

---

## 基本カラム定義

### scenario_id
Step4 以降で使用しているシナリオIDをそのまま記載する。  
再採番しないこと。

### scenario_name
Step5 で定義したシナリオ名を記載する。

### priority
Step6 で付与した優先順位を記載する。  
値は `Crit / High / Mid / Low` のいずれかとする。

### preconditions
テスト実行前提条件を記載する。  
複数条件がある場合は、CSVセル内で区切りを統一すること。

### steps
時系列手順を記載する。  
複数手順を1セルに格納する場合は、区切り方法を統一すること。

### expected_results
期待値を記載する。  
手順対応が必要な場合は、steps と対応が分かるように整形すること。

### test_data
入力データ、ユーザー条件、状態条件などを記載する。  
複数データ条件がある場合は、CSVセル内で区切りを統一すること。

---

## 出力形式の基本方針

### 1. 1行 = 1テストケース
原則として、1つの `scenario_id` に対してCSVの1行を割り当てること。

### 2. 複数要素はセル内で表現する
`preconditions`、`steps`、`expected_results`、`test_data` に複数要素がある場合でも、原則として列を増やさず、1セル内に整形して格納する。

### 3. カラム意味を変えない
CSVへ落とすために、元の項目意味を変えたり、省略しすぎたりしないこと。

### 4. 既存内容を勝手に要約しない
CSVに収めるために内容を勝手に削ったり、意味が変わるほど短縮したりしないこと。

---

## セル内の整形ルール

### preconditions
複数条件がある場合は、以下のいずれかで統一する。

- `1. ... / 2. ... / 3. ...`
- `・... | ・... | ・...`

同一成果物内で混在させないこと。

### steps
複数手順がある場合は、番号付きで1セル内に格納すること。

望ましい例:
- `1. 申請登録画面を開く 2. 必須項目を入力して登録する 3. 承認者が承認する`
- `1) 申請登録画面を開く / 2) 必須項目を入力して登録する / 3) 承認者が承認する`

`steps` と `expected_results` の対応が必要なため、手順番号は可能な限り保持すること。

### expected_results
`steps` に対応する期待値が分かるように整形すること。

望ましい例:
- `1. 申請登録画面が表示される 2. 申請データが登録され、状態が「申請中」となる 3. 承認後、状態が「承認済」に更新される`

### test_data
条件ごとに区切って記載すること。

望ましい例:
- `申請者=user_a / 承認者=approver_b / 申請区分=新規 / 申請金額=100000`
- `対象状態=差戻し / 外部連携結果=タイムアウト`

---

## CSV文字列整形ルール

### 1. カンマを含むセルは引用符で囲む
セル値にカンマ `,` を含む場合は、CSVとして列崩れしないようにダブルクォートで囲むこと。

### 2. 改行を含む場合の扱いを統一する
CSVセル内に改行を残すか、スペースや区切り記号へ変換するかを統一すること。

基本方針としては、ユーザーから明示指定がない限り、  
**改行はセル内区切り記号へ変換し、1レコード1行を維持しやすい形式** を優先する。

### 3. ダブルクォートを含む場合はエスケープする
セル値にダブルクォート `"` を含む場合は、CSVルールに従って `""` としてエスケープすること。

### 4. 区切り記号を統一する
セル内の複数要素を区切る場合、同一CSV内で表現を混在させないこと。

### 5. 不要な装飾を入れない
Markdownの表記、見出し、箇条書き記号、HTMLタグなどを残さないこと。  
CSVは飾り気がない方が健全。

---

## 推奨するセル内表現

ユーザー指定がない場合、以下を推奨形式とする。

- `preconditions`: `1. ... / 2. ... / 3. ...`
- `steps`: `1. ... / 2. ... / 3. ...`
- `expected_results`: `1. ... / 2. ... / 3. ...`
- `test_data`: `項目A=... / 項目B=... / 項目C=...`

この形式なら、CSVとして1行に載せやすく、かつ手順対応も少しだけ読みやすい。

---

## 出力方法

出力は、原則としてCSVそのものとして提示する。  
見出し行を含め、CSVとしてそのまま保存できる形式で出力すること。

基本形:

```csv
scenario_id,scenario_name,priority,preconditions,steps,expected_results,test_data
SCN-001,"新規申請の登録から承認完了",High,"1. 申請者ユーザーでログイン可能である / 2. 対象申請データが未登録である","1. 申請登録画面を開く / 2. 必須項目を入力して登録する / 3. 承認者が承認する","1. 登録画面が表示される / 2. 申請データが登録され状態が申請中となる / 3. 承認後に状態が承認済へ更新される","申請者=user_a / 承認者=approver_b / 申請区分=新規 / 金額=100000"
````

---

## 補助列の扱い

以下のような補助情報は、最終CSVの基本カラムには含めない。

* `related_business_flow_ids`
* `related_viewpoint_ids`
* `notes`
* `priority_rationale`

ただし、ユーザーがレビュー用または管理用として追加列を希望する場合は、別版として含めてよい。

例:

* `priority_rationale`
* `related_viewpoint_ids`

その場合も、ユーザー指定がない限り、基本版と拡張版を混同しないこと。

---

## このステップでやってはいけないこと

このステップでは、以下を行ってはならない。

* 新しいテストケースを追加すること
* scenario_id を変更すること
* priority を再評価すること
* expected_results の意味を変えること
* 内容を勝手に要約・削除すること
* Markdown表や説明文をCSV本体に混在させること

このステップは整形工程であり、再設計工程ではない。

---

## 品質チェック観点

出力前に、少なくとも以下を確認すること。

### 1. 指定カラム順になっているか

ユーザー指定のカラム順、または基本カラム順に従っているか。

### 2. 1行1ケースになっているか

ケースが複数行に分裂していないか。

### 3. 列崩れの原因がないか

カンマ、改行、引用符の扱いに問題がないか。

### 4. 手順と期待値の対応が読めるか

`steps` と `expected_results` の番号や順序が崩れていないか。

### 5. 値の欠落がないか

Step6 までに確定した値が落ちていないか。

### 6. 不要な装飾が残っていないか

Markdown、箇条書き記号、見出しなどが混入していないか。

---

## このステップの完了条件

以下を満たしたら、このステップは完了とみなす。

* 指定カラム構成でCSV出力用データが作成されている
* すべてのテストケースが1行ずつ出力されている
* `steps` と `expected_results` がセル内で整形されている
* CSVとして列崩れしない形式になっている
* Step6 までの確定内容が保持されている

---

## ステップ完了時の付帯出力

CSV本体に加えて、必要に応じて以下を簡潔に添えてよい。

### 確定事項

出力対象件数、採用カラム構成、セル内整形ルールを要約する。

### 要確認事項

CSV列追加要否、改行保持要否、補助列出力要否など、運用上確認したい点があれば列挙する。

### レビュー依頼

CSVとして利用しやすい形式になっているか、列構成やセル内表現に問題がないかを確認依頼する。

ただし、CSV本体と説明文は明確に分けること。
