# issue_log_template.md

## 目的

本テンプレートは、シナリオテスト設計運用の各ステップで見つかった未確定事項、資料不足、資料間矛盾、解釈保留事項、運用判断待ち事項を記録・管理するためのものである。

本ログの目的は、未確定事項を曖昧なまま後続工程へ持ち込むのではなく、  
「何が未確定で、どこに影響し、どう扱うべきか」を明示することである。

本ログは、AIエージェントが不明点を勝手に補完しないための抑止装置でもある。

---

## 利用方針

- 各ステップで見つかった `要確認事項` を、本ログへ記録する
- 軽微な表現差ではなく、後続工程や成果物の妥当性に影響するものを優先して記録する
- 同じ論点が複数ステップで再登場した場合は、既存 issue を更新する
- issue が未解決のまま後続工程へ影響する場合は、影響範囲を明示する
- 解消済みになった issue も、履歴のために削除せず状態更新で残す

---

## issue の分類例

必要に応じて、以下のような分類を使ってよい。

- `資料不足`
- `資料間矛盾`
- `前提不明`
- `スコープ不明`
- `業務解釈保留`
- `用語不明`
- `権限不明`
- `状態遷移不明`
- `期待値根拠不足`
- `データ条件不明`
- `優先度判断保留`
- `その他`

分類は増やしてよいが、細かく分けすぎないこと。

---

## ステータス定義

各 issue には、少なくとも以下のいずれかのステータスを付与する。

- `Open`  
  未解決。今後の確認・判断が必要な状態。

- `In Review`  
  ユーザー確認中、または判断待ちの状態。

- `Resolved`  
  解決済み。判断または追加資料により確定した状態。

- `Accepted Risk`  
  未解決だが、現時点ではその前提で進めることを受容した状態。

- `Out of Scope`  
  検討対象外であると確定した状態。

---

## 優先度の考え方

必要に応じて、issue 自体にも優先度を付けてよい。

- `High`  
  後続工程へ大きく影響する。誤ると観点・シナリオ・期待値が崩れる。

- `Mid`  
  影響はあるが、局所的。暫定対応で進められる可能性がある。

- `Low`  
  影響は限定的。後で確認しても致命的ではない。

issue の優先度はテストケースの priority とは別物であるため、混同しないこと。

## 記録項目

各 issue には、少なくとも以下を含めること。

- `issue_id`
- `title`
- `category`
- `detected_in_step`
- `description`
- `impact_scope`
- `related_artifacts`
- `related_documents`
- `proposed_action`
- `status`
- `owner`
- `notes`

必要に応じて、以下を追加してよい。

- `severity`
- `due`
- `resolved_by`
- `resolution_summary`

## 各項目の記述ルール

### issue_id
一意のIDを付与する。  
形式は `ISS-001` のようにする。

### title
issue の内容が短く分かるタイトルを付ける。

望ましい例:
- 承認権限の条件が資料間で不一致
- 差戻し後の再提出可否が不明
- 外部連携失敗時の状態遷移定義が不足

避けるべき例:
- 要確認
- 不明点
- 課題
- 気になるところ

### category
issue の分類を記載する。

### detected_in_step
どのステップでこの issue を検出したかを記載する。  
例: `Step1`, `Step3`, `Step5`

### description
何が不明・矛盾・未確定なのかを具体的に記載する。

### impact_scope
この issue がどの工程や成果物に影響するかを記載する。

例:
- Step2 の観点抽出に影響
- Step4 の異常系シナリオ境界に影響
- Step5 の expected_results 定義に影響

### related_artifacts
関連する成果物IDや項目を記載する。

例:
- `VP-003`
- `BF-005`
- `SCN-008`
- `SCN-012 expected_results`

### related_documents
根拠となる資料名、または矛盾している資料名を記載する。

### proposed_action
どう対応すべきかの案を記載する。

例:
- ユーザーへ承認条件の優先資料を確認する
- 一旦「要確認」として異常系詳細化を保留する
- Step2 観点一覧の見直し候補とする

### status
現在の状態を記載する。

### owner
誰が確認・判断する想定かを記載する。

例:
- ユーザー
- 業務担当
- QA設計担当
- 未定

### notes
補足事項を記載する。

## 出力フォーマット

以下の形式を基本とする。

```markdown
# Issue Log

| issue_id | title | category | detected_in_step | description | impact_scope | related_artifacts | related_documents | proposed_action | status | owner | notes |
|---|---|---|---|---|---|---|---|---|---|---|---|
| ISS-001 | ... | ... | Step1 | ... | ... | ... | ... | ... | Open | ユーザー | ... |
| ISS-002 | ... | ... | Step3 | ... | ... | ... | ... | ... | In Review | 業務担当 | ... |
````

必要に応じて、Open / Resolved などのセクションに分けてもよい。

## 記載例

```markdown
# Issue Log

| issue_id | title | category | detected_in_step | description | impact_scope | related_artifacts | related_documents | proposed_action | status | owner | notes |
|---|---|---|---|---|---|---|---|---|---|---|---|
| ISS-001 | 承認権限の条件が資料間で不一致 | 資料間矛盾 | Step1 | 要件定義書では部門長のみ承認可能と読める一方、画面仕様書では代理承認者の存在が示唆されている | Step2 の権限観点、Step4 の正常系/異常系シナリオ、Step5 の expected_results に影響 | VP-004, BF-006, SCN-003 | 要件定義書 3.2節 / 画面仕様書 承認画面 | 承認可能ロールの正式定義をユーザーへ確認する | Open | ユーザー | 代理承認の有無でシナリオ分割が必要になる可能性あり |
| ISS-002 | 差戻し後の再提出回数制限が不明 | 前提不明 | Step3 | 差戻し後に何度でも再提出できるか、回数制限があるかが資料から判断できない | Step4 の異常系定義、Step5 の preconditions と expected_results に影響 | BF-008, SCN-007 | 業務フロー図 / 運用手順書 | 一旦回数制限なし前提で進めるか確認する | In Review | 業務担当 | Accepted Risk にする場合は明記が必要 |
```

## 記録対象の判断基準

### 記録すべきもの

以下に該当するものは、原則として記録する。

* 資料を読んでも仕様や業務ルールが確定できない
* 資料同士で矛盾している
* 後続工程で観点・フロー・シナリオ・期待値に影響する
* 優先度判断に影響する
* スコープや対象外の境界に関わる

### 記録しなくてもよいもの

以下のようなものは、毎回 issue 化しなくてもよい。

* 文言の軽微な表記ゆれ
* 後続成果物に影響しない細かい説明不足
* レビュー時に即時解消済みの軽微な修正
* 単なる言い換えで済むもの

---

## 運用ルール

### 1. issue を隠さない

未確定事項を成果物本文から消してはいけない。
必要に応じて本文と issue log の両方に記載する。

### 2. 同じ issue を乱立させない

同一論点は、新規 issue を増やすのではなく既存 issue を更新することを優先する。

### 3. Resolved でも履歴を残す

解決した issue も削除しない。
状態更新と解決内容の追記で管理する。

### 4. Accepted Risk は明示する

未解決のまま進める場合は、それを黙ってやらない。
Accepted Risk として残す。

### 5. ステップ成果物と切り離しすぎない

issue log は独立管理しつつ、関連成果物IDとの接続を保つこと。

---

## AIエージェント向けルール

* 不明点を見つけたら、勝手に埋めず、issue 候補として扱うこと
* その issue が後続工程へ影響するなら、impact_scope を必ず書くこと
* 修正により issue が解消した場合は、新規起票ではなく既存 issue の status 更新を優先すること
* issue が残っている状態で次工程へ進む場合は、その前提を明示すること
